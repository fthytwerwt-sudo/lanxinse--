# 直播切片双路线产品总方案

状态：产品设计合同已创建，待分阶段实现
事实基线：GitHub main d12f8457761cc27933318ebd1222e316394774d7
规划日期：2026-07-20

## 1. 主结论

本产品必须是一个入口、两条主路线、一个审核与交付出口：

1. Route A（路线 A）处理运动＋讲解型直播，以自动生成分类视频、人工抽检为目标。
2. Route B（路线 B）处理纯口播型直播，以完整语义解析、候选生成、人工确认后正式剪辑为硬流程。
3. mixed（混合型）不得整条强塞进 A 或 B，必须先按时间段拆分，再分别处理。
4. unknown（无法确认）不得自动生产正式视频，必须进入人工路由确认。
5. 飞书只承载查看、通知和审核交互，不是项目事实源；GitHub main 继续承载可审计的项目事实。

产品主链：

    素材进入
    → 技术预检
    → material_route（素材处理路线）判断
    → Route A / Route B / mixed 拆分 / unknown 人工路由
    → analysis_unit（解析单元）
    → candidate_clip（候选切片）或 Route A 自动输出候选成片
    → review_task（人工审核任务）
    → 修改、重切或通过
    → final_clip（正式视频）
    → 飞书交付查看
    → 人工确认交付
    → execution_record（执行记录）与 GitHub 项目事实回写

## 2. 产品目标与边界

### 2.1 本合同要解决

- 将两种素材从入口开始分流，禁止继续用同一动作闸门覆盖全部直播。
- 给后续 Codex 提供可逐阶段实现的模块、字段、状态、闸门、失败标准和最小验证。
- 建立 AI、Codex、人工、飞书与 GitHub 的职责边界。
- 将技术通过、内容通过、人审通过、业务通过和正式交付通过拆开记录。

### 2.2 本轮明确不做

- 不编写或修改正式业务代码。
- 不运行任何直播素材，不生成新视频。
- 不调用 ASR、视觉模型或飞书 API。
- 不创建飞书应用，不填写 app_id 或 app_secret。
- 不安装依赖。
- 不引入 LangGraph、LangChain、复杂 runtime、Agent orchestration、RAG 或数据库。
- 不把方案写成当前已经实现的能力。

## 4. 两条路线事实差异表

| 维度 | Route A 运动＋讲解型 | Route B 纯口播型 |
|---|---|---|
| 主要锚点 | 可命名动作、身体部位、动作循环、讲解与动作同题 | 语义主题、观点链、问答链、销售链、故事链、完整表达边界 |
| 主要输入证据 | 视觉动作单元＋完整 ASR＋动作口令 | 完整 ASR＋说话人/互动线索＋轻量画面连续性 |
| 视觉重要性 | 高；决定是否是真动作、同一动作、动作完整 | 低到中；只检查跳帧、遮挡、人物连续性，不把普通手势当动作锚点 |
| ASR 重要性 | 高；用于回查用途、问题、原因、方法、边界 | 最高；决定主题、结构、起止点、重复和噪音 |
| 自动切片能力 | 目标为通过硬闸门后自动组装分类视频 | 只能先生成候选；正式视频必须经人工确认 |
| 人工审核比例 | 自动通过项抽检，低置信度全审 | 初期 100% 候选人审，稳定后再讨论抽检 |
| 分类方式 | 讲解与动作的结构关系 | 语义内容结构，不使用动作分类 |
| 主要风险 | 口播动作错配、动作不完整、身体部位冲突、跳题 | 半句话、伪完整结构、关键词误判、重复、错误卖点/证据、切点不自然 |
| 通过标准 | 同动作＋完整解释链＋动作循环＋无确定性冲突 | 语义结构完整＋边界自然＋上下文自足＋人工确认 |
| 当前成熟度 | 部分成立：规则与单素材链存在，通用性待验证 | 待创建：动作路线在该素材上的严格盲测已失败 |

## 5. 产品总架构

### 5.1 共享入口层

输入是 source_asset（原始素材）。共享入口只做：

- 文件存在、格式、音轨、时长、散列、来源和访问权限检查。
- 建立 job（处理任务）与 source_asset 编号。
- 记录原始素材，不在 Git 中提交媒体。
- 检查是否已有相同 source_hash，避免重复处理。

### 5.2 路由层

输出 material_route（素材处理路线）、route_confidence（路线置信度）、route_evidence（路线证据）和 route_decision_status（路线决定状态）。

- motion_explanation（运动＋讲解型）：有目的性动作、动作身份和同题讲解证据。
- speech_only（纯口播型）：语义表达为主体，没有稳定可教学动作链。
- mixed（混合型）：不同时间段分别满足 A 与 B。
- unknown（无法确认）：证据不足、模型失败、两类证据冲突或置信度未达经验证门槛。

任何自动判断都允许人工 override（覆盖），但必须保留原决定、覆盖人、覆盖原因和新版本。

### 5.3 路线处理层

- Route A：视觉动作发现优先，完整 ASR 同动作回查，确定性闸门，结构组装，自动候选成片，人工抽检。
- Route B：完整 ASR 优先，语义单元切分，内容角色标注，结构候选生成，评分与去重，人工审核，批准后正式剪辑。
- mixed：先生成不重叠时间分区，每个分区独立继承 A 或 B；跨区候选默认禁止，除非人工明确合并。
- unknown：只生成路由审核任务，不生成正式候选视频。

### 5.4 共享审核与交付层

两条路线统一进入 candidate_clip、review_task、feedback（人工反馈）、final_clip 和 execution_record。

飞书镜像任务和审核状态；系统记录操作流水；审核通过后生成正式视频；最终形成可提交 Git 的最小事实包。

### 5.5 产品界面与操作流程

以下是产品界面合同，不代表当前已有 Web UI。第一阶段允许用本地索引、命令行结果和飞书审核台承载同一信息结构；是否建设独立前端由后续阶段验收决定。

| 页面 / 工作台 | 主要使用人 | 必须显示 | 允许操作 | 权限与失败保护 |
|---|---|---|---|---|
| 任务创建页 | 操作员 | 素材来源、文件散列、时长、权限检查、重复素材提示 | 登记素材、补充业务标签、提交处理 | 无读取权限或散列冲突时禁止开始；不上传媒体到 Git |
| 路由结果页 | 操作员、审核人 | material_route、route_confidence、route_evidence、分段时间线、规则/模型版本 | 接受路线、人工 override、mixed 分段修正、转 unknown | override 必填原因并保留原决定；证据不足不得伪装为高置信度 |
| 处理详情页 | 操作员 | 当前阶段、输入/输出版本、步骤状态、失败证据、重试次数 | 重试可重入步骤、终止任务、打开失败详情 | 重试必须使用幂等键；不得覆盖上一版本证据 |
| 候选列表页 | 剪辑审核人 | 路线、分类、时间范围、评分、风险标签、重复组、代理预览入口 | 筛选、排序、批量分派、打开单条审核 | Route B 不提供“自动正式通过”；失效链接显示明确错误 |
| 单条审核页 | 审核人 | 视频/封面入口、ASR 上下文、语义或动作证据、切点、当前版本、历史反馈 | approve、reject、request_recut、改分类、改切点、补反馈 | 过期版本禁止提交；终态动作须二次确认并记录 reviewer_id |
| 正式交付页 | 业务负责人 | final_clip、审核链、文件散列、下载/播放入口、到期时间、业务状态 | 业务确认、退回、下载、复制受控链接 | 人工审核通过不等于业务通过；权限或链接到期时不得标记 delivered |
| 失败与重试页 | 操作员、维护人 | failure_code、失败阶段、错误摘要、可重试性、下一步、责任人 | 重试、转人工、切换 fallback、关闭失败 | secret 和原始敏感响应必须脱敏；不可恢复失败不得无限重试 |

统一操作顺序为：创建任务 → 查看/修正路由 → 跟踪路线处理 → 审核候选 → 生成正式视频 → 业务确认与交付。每个界面只展示其权限范围内的数据，飞书页面是系统状态的操作镜像，不是唯一事实源。

## 6. 核心产品实体

首次出现的统一实体及中文含义：

- job：一次完整处理任务。
- source_asset：输入的原始直播素材。
- material_route：素材进入的处理路线。
- analysis_unit：ASR、画面或语义解析的最小单元。
- candidate_clip：等待系统或人工确认的视频候选。
- review_task：需要人工处理的审核任务。
- final_clip：审核后正式输出的视频。
- feedback：人工修改、退回和补充意见。
- execution_record：执行、模型、版本、失败证据和重试记录。

详细字段、关系、状态机和幂等规则见 04 号文件。

## 7. AI、Codex、人工、飞书、GitHub 分工

| 角色 | 允许负责 | 禁止越权 |
|---|---|---|
| AI | ASR、视觉观察、语义角色、候选分类、证据与置信度 | 不得单独宣布动作专业、健康效果、审美、业务或发布通过 |
| Codex / 普通 Python 模块 | 调度显式步骤、验证输入、执行确定性规则、生成索引、可重跑剪辑、记录证据 | 不得猜核心分类标准、自动放宽阈值、把失败候选改写为完成 |
| 人工审核人 | 路由覆盖、分类修正、切点修改、内容完整性、发布适合度 | 不得在没有证据时把系统事实改成已实现能力 |
| 业务负责人 | 最终审美、合规、健康表达、业务和正式交付确认 | 不能用口头通过覆盖缺失的技术证据；需留下记录 |
| 飞书 | 通知、列表、预览入口、按钮和审核意见 | 不是项目事实源，不保存唯一正式状态 |
| GitHub main | 规划、规则、索引、执行报告、版本和完成证据 | 不存视频、密钥、ASR 缓存和临时链接 |

## 9. 统一判断标准

| 判断层 | 通过定义 | 绝不能偷换为 |
|---|---|---|
| 技术通过 | 文件存在、元数据可读、有所需音视频流、完整解码 | 内容通过 |
| 语义解析通过 | 证据跨度、角色、主题和边界均可追溯，且非关键词单点命中 | 候选切片通过 |
| 动作配对通过 | 同动作/同问题/同目的、动作身份和部位一致、循环完整、无跳题 | 动作专业或健康效果通过 |
| 候选切片通过 | 结构完整、切点自然、无关键上下文缺失，进入审核池 | 剪辑师直接可用 |
| 剪辑师直接可用 | Route A 通过自动硬闸门且达到已验证质量阈值 | 用户或业务通过 |
| 人工审核通过 | 指定审核人对当前版本明确 approved | 审美和业务通过 |
| 审美 / 人感通过 | 业务指定人员确认观看体验和表达自然 | 技术通过 |
| 业务通过 | 内容、合规、品牌、健康表达和发布价值被负责人确认 | 本地可播放 |
| 正式交付通过 | final_clip 可访问、来源可追溯、审核和接收均确认 | 仅上传成功 |
| 批量稳定运行 | 跨多素材达到预先确认的质量、失败率、重试和人审负担门槛 | 单素材成功 |

任务级完成态必须与视频数量解耦。尤其：

    candidate 或 plan 数量大于 0
    且 editor_ready_direct 等于 0
    → 不得写 completed
    → 只能写 review_only_result 或 blind_test_failed_with_evidence

## 10. 失败与降级总原则

- 路由不清：进入 unknown，不强行选择。
- ASR 失败：阻断 Route B；Route A 若必须依赖解释链也阻断，不用视觉结果代替。
- 视觉失败：Route A 进入人工动作复核；Route B 可继续语义候选，但不得宣布画面通过。
- 动作不清：不得创建自动直出动作视频。
- 语义不清：保留上下文候选或进入人工复核，不靠关键词补齐。
- 切点不确定：生成可调 start/end 的 review_task，不直接正式输出。
- 飞书上传失败：保留系统任务并改用外部签名链接；不得丢审核记录。
- 飞书权限不足：blocked_feishu_permission_missing，转人工管理员处理。
- 人工长期未处理：只提醒和升级负责人，不自动通过。
- Git 事实同步失败：交付可以保持 delivered_pending_fact_sync，但不能写正式项目闭环完成。

## 11. 成功标准

产品未来只有在以下条件逐层满足时，才可以从“规划合同”升级为“已实现产品”：

1. 路由对 motion_explanation、speech_only、mixed、unknown 均有验证样本。
2. Route A 在非 5 月 13 日素材上通过硬闸门和人工抽检。
3. Route B 不再使用动作类别，候选结构和切点经人工验证。
4. 统一实体、状态和 feedback 回流均有测试。
5. 飞书账号 probe 证明权限、卡片回调、链接访问和审核回写可用。
6. 正式视频与审核记录、来源候选、执行版本一一对应。
7. GitHub main 可回读最小事实包，但媒体和 secret 不进入 Git。
