# 阶段实施路线图与验收矩阵

状态：实施合同已设计，尚未开始业务实现
原则：按依赖顺序推进，不给未经验证的日历工期

## 1. 阶段总览

| 阶段 | 名称 | 核心结果 | 是否接飞书 |
|---|---|---|---|
| Phase 1 | 素材路由＋共享输入层 | 四类 material_route、统一 ID、技术预检 | 否 |
| Phase 2 | 运动＋讲解型产品化 | 通用 Route A 候选成片和抽检 | 否 |
| Phase 3 | 纯口播解析＋人工审核 | 语义候选、版本、审核和正式输出 | 否 |
| Phase 4 | 飞书审核台试点 | 四张表、卡片、回调和存储 probe | 是，试点 |
| Phase 5 | 统一产品整合 | 一个 job、两路线、统一状态和反馈 | 是 |
| Phase 6 | 批量素材验证 | 跨素材质量、失败率和人审负担证据 | 是 |

进入下一阶段只依据验收，不依据“代码写完”或日历时间。

## 2. Phase 1：素材路由＋共享输入层

### 2.1 目标

- 建 job、source_asset、material_route、route_segment、execution_record 的最小模型。
- 实现技术预检、重复素材识别、四类路由建议和人工覆盖。
- 初期所有路由建议人工确认。

### 2.2 前置依赖

- 四类路由的人工标注样本。
- 用户确认 mixed 分段审核方式。
- 路由字段、状态、版本和 unknown 降级合同锁定。

### 2.3 输入

- 代表性 motion_explanation、speech_only、mixed、unknown 素材或可用证据。
- 人工路线金标准与原因。
- 文件访问、音轨和技术元数据。

### 2.4 输出

- job/source_asset manifest。
- technical_probe。
- material_route 建议、置信度、证据和版本。
- route_segment 时间轴。
- 路由人工复核清单。
- execution_record 和失败证据。

### 2.5 AI 工作

- 提供视觉动作、语义主体和 mixed 时间段证据。
- 不决定最终路由阈值。

### 2.6 Codex 工作

- 实现显式 schema、路由规则、版本、幂等和 unknown 降级。
- 生成可追溯报告和测试。

### 2.7 人工工作

- 标注金标准。
- 逐条确认路由建议和 mixed 边界。
- 确认未来自动阈值，未确认前保持人工。

### 2.8 验收

- 四类均有正例和反例。
- unknown 不被强行归类。
- mixed 段覆盖时间轴、无 A/B 重叠。
- 人工覆盖生成新版本并使旧候选失效。
- 同一输入重跑不重复建 job。
- 技术、路由和人工状态分开。

### 2.9 失败标准

- 普通手势使纯口播被大量误路由到 A。
- 路由只返回 A/B，没有 mixed/unknown。
- 固定阈值未经样本验证就自动继续。
- 人工覆盖会覆盖旧证据或无法追溯。
- 缺路由金标准却宣称准确。

### 2.10 是否进入下一阶段

只有 Phase 1 验收通过，Phase 2/3 才能消费 material_route；否则 blocked_route_contract_not_validated。

## 3. Phase 2：运动＋讲解型产品化

### 3.1 目标

- 将现有动作主题规则从单一 5 月 13 日专用实现拆为通用 Route A。
- 生成可分类的 candidate_clip，并建立人工抽检。

### 3.2 前置依赖

- Phase 1 路由通过。
- 至少一条非 5 月 13 日人工确认运动＋讲解型素材。
- 动作术语/别名、健康风险和抽检责任人。

### 3.3 输入

- confirmed motion_explanation source_asset。
- 完整 ASR 与视觉证据。
- 动作分类和确定性冲突合同。

### 3.4 输出

- action_topic 主表和证据。
- content_classification、structure_type、assembly_mode。
- candidate_clip、技术探针和 review_task。
- logic_mismatch、partial、manual_review 失败清单。
- approved 后的 final_clip。

### 3.5 AI 工作

- 视觉动作观察、动作身份候选、口播角色建议。
- 不覆盖确定性冲突。

### 3.6 Codex 工作

- 完整 ASR 回查、同动作聚合、确定性闸门、幂等组装和技术验证。
- 移除专用 TOPIC_SPECS 作为通用入口。

### 3.7 人工工作

- 审核动作身份、内容连贯性、切点、人感、健康表达和业务价值。
- 提供误放行/误杀反馈。

### 3.8 验收

- 非 5 月 13 日素材能通用运行。
- 普通手势、部位冲突、动作名冲突和跳题反例均不会直出。
- 自动候选全部有原始时间码和证据。
- 01 至 06 内容分类与 07 组装方式、08 审核状态分层。
- 技术通过不自动写内容、人审或业务通过。

### 3.9 失败标准

- 使用素材专用动作 ID、口播句子或人工时间码。
- candidate 数量大于 0 就写 completed。
- editor_ready_direct 为 0 仍写路线成功。
- 为增加数量放宽同题、循环或冲突闸门。
- 批量未验证就宣称稳定。

### 3.10 是否进入下一阶段

Route A 可独立完成最小闭环后可与 Phase 3 并行迭代，但不得跳到 Phase 5 整合。

## 4. Phase 3：纯口播解析＋人工审核

### 4.1 目标

- 建 ASR 优先语义层、纯口播候选分类、评分去重、重切版本和 100% 人审。

### 4.2 前置依赖

- Phase 1 路由通过。
- 纯口播人工金标准覆盖主要分类和反例。
- 用户确认试点候选时长、审核人和业务风险规则。

### 4.3 输入

- confirmed speech_only source_asset。
- 完整 ASR、时间码和质量状态。
- 语义字段、13 类候选合同、正负样本和人工切点。

### 4.4 输出

- topic 和 sentence_turn analysis_unit。
- semantic_role 证据。
- candidate_clip、评分分项、duplicate cluster。
- review_task、feedback 和 candidate_version。
- approved 后 final_clip。

### 4.5 AI 工作

- topic、语义角色、结构链、候选理由和置信度。
- 不对案例真实性、健康效果、优惠有效性或业务通过盖章。

### 4.6 Codex 工作

- 完整性检查、相邻合并、去重、排序、候选视频、版本和反馈应用。
- 阻止关键词单点命中升级为结构完成。

### 4.7 人工工作

- 100% 审核分类、边界、上下文、风险和适合发布性。
- 修改开头/结尾、改分类或要求重新解析。

### 4.8 验收

- 动作循环不再是 Route B 主闸门。
- 至少覆盖问答、知识、观点、问题解决、销售/CTA、互动、重复和噪音样本。
- question/answer、objection/answer、卖点/证据均验证对应性。
- 重切生成新版本，不覆盖旧候选。
- final_clip 指向准确 approved candidate_version。
- 人审指标可统计：分类修改率、切点修改率、退回率、审核负担。

### 4.9 失败标准

- 仍输出动作分类。
- 完整 ASR 被写成完整语义解析已完成。
- 只凭关键词判断结构。
- 37 条技术视频被当作 Route B 成功样本。
- 候选未经人工就成为正式视频。
- 审核结果串到错误版本。

### 4.10 是否进入下一阶段

只有 Route B 最小离线闭环通过，才允许进入 Phase 4 飞书试点；否则飞书只会放大错误。

## 5. Phase 4：飞书审核台试点

### 5.1 目标

- 用企业自建应用、应用机器人、卡片和四张多维表格完成少量审核试点。
- 验证 F3 主路线与 F2 降级，不承诺 F1。

### 5.2 前置依赖

- Phase 3 review_task/feedback/candidate_version 合同通过。
- 企业管理员、试点审核人、权限范围和批准存储方案。
- 回调安全、幂等和失败兜底设计。

### 5.3 输入

- 少量 A/B 候选和审核代理视频。
- 飞书测试应用与最小权限。
- 小于 20 MB、大于 20 MB、代表候选和代表高清测试文件。

### 5.4 输出

- 处理任务、候选审核、正式交付、失败重试四张表。
- 新任务通知和审核卡片。
- 五类审核操作回写。
- F1/F2/F3 上传、打开、预览、权限和成本 probe 报告。

### 5.5 AI 工作

- 只生成卡片所需摘要、理由和风险提示。

### 5.6 Codex 工作

- 实现镜像同步、3 秒快速响应、异步处理、事件幂等、串行 Base 写和存储降级。

### 5.7 人工工作

- 管理员审批权限。
- 在 PC、iOS/Android 真实审核。
- 验证外部用户隔离、下载权限、链接过期和撤权。

### 5.8 验收

- 应用机器人卡片可正确触发通过、退回、重切、分类错误和意见。
- 重复点击不重复执行；旧版本按钮被拒绝。
- 卡片失败有重新提交/详情页兜底。
- 同表写串行，Write conflict 可恢复或正确阻断。
- F3 可查看审核代理、访问正式高清并保持映射。
- 飞书不可用时 F2 可继续审核。
- 不写“卡片内直接播放视频”或“预览稳定通过”除非实测。

### 5.9 失败标准

- 使用群自定义机器人承担审核回传。
- 卡片点击直接修改 Git 或 final_clip。
- 卡片超时后依赖飞书自动补推。
- Base 成为唯一状态源。
- 未验证权限、容量或设备体验就宣布可上线。
- secret 进入 Git、表格或日志。

### 5.10 是否进入下一阶段

管理员、卡片回调、四张表、F3/F2 和设备 probe 均通过才进入 Phase 5。

## 6. Phase 5：统一产品整合

### 6.1 目标

- 一个入口、两路线、mixed/unknown、一个审核和交付出口。
- 统一状态机、失败恢复、飞书镜像和 Git 事实包。

### 6.2 前置依赖

- Phase 1 至 4 的各自最小闭环均通过。
- 状态、字段、错误类型和反馈映射冻结为版本化合同。

### 6.3 输入

- A/B 经过验证的模块。
- 飞书试点结果。
- 统一 schema 和状态迁移表。

### 6.4 输出

- 统一 job 工作目录/manifest。
- route adapter、candidate adapter 和 review adapter。
- 统一 final_clip 和 delivery 记录。
- 统一 execution_record、失败重试和 Git 事实同步。

### 6.5 AI 工作

- 按 route adapter 运行对应分析，不跨路线混用分类。

### 6.6 Codex 工作

- 组合普通 Python 模块、任务清单和幂等步骤。
- 不建设 LangGraph、LangChain 或复杂 runtime。

### 6.7 人工工作

- 验证一个界面能区分不同路线、状态、版本和风险。
- 确认业务状态词和上线边界。

### 6.8 验收

- route A/B/mixed/unknown 均走统一 job 但不共享错误算法。
- 所有 feedback 回到正确层。
- 技术、语义、动作、候选、人审、审美、业务、交付、Git 状态独立。
- 可从 final_clip 回溯 route、candidate_version、source_ranges 和执行版本。
- 重跑和重复回调不产生重复正式视频。

### 6.9 失败标准

- 为统一产品而把 A/B 合成一个分类器。
- 单一 completed 覆盖所有状态轴。
- mixed 跨段自动拼接。
- 飞书状态覆盖系统状态。
- 任何一步不可追溯或不可重跑。

### 6.10 是否进入下一阶段

通过集成回归后才进入 Phase 6，不以“界面能打开”作为整合完成。

## 7. Phase 6：批量素材验证

### 7.1 目标

- 用多来源、多时长、多画质和四类路线素材证明质量、稳定性和人工负担。

### 7.2 前置依赖

- Phase 5 全链技术和状态回归通过。
- 用户确认批量样本集、业务验收指标和停止线。

### 7.3 输入

- 代表真实分布的素材集。
- 路由、分类、切点和业务金标准。
- 失败恢复与人工审核容量。

### 7.4 输出

- 路由混淆矩阵。
- Route A 误放行/误杀和抽检结果。
- Route B 分类、切点、去重和审核负担指标。
- mixed 分段误差。
- 飞书/存储可用率、回调失败和重试统计。
- Git 事实闭环与失败清单。

### 7.5 AI 工作

- 批量提供证据和候选，不越过人工/业务闸门。

### 7.6 Codex 工作

- 批量可重跑、限流、失败隔离、报告和事实落库。

### 7.7 人工工作

- 按预先抽样方案复核。
- 判断审美、人感、健康、业务和发布。

### 7.8 验收

- 指标门槛必须由用户在实施前确认，不由 Codex生成。
- 单个任务失败不会污染其他任务。
- 可重试和不可重试错误正确区分。
- 人工负担在公司可接受范围。
- 多批次结果可复现，执行版本可追溯。

### 7.9 失败标准

- 只报告平均值，不保留失败样本。
- 因候选少而调低硬闸门。
- 未统计人工修改率和审核时长。
- 单素材结论外推到批量稳定。
- 未通过业务人审就写正式上线。

### 7.10 是否允许产品上线

只有用户明确确认批量技术、内容、人审、审美/人感、业务和交付标准后，才能进入生产；Codex 不自行决定。

## 8. 分层验收矩阵

| 验收层 | 必需证据 | 失败标准 | 决定人 |
|---|---|---|---|
| 技术通过 | 文件、流、完整解码、hash | 缺流/损坏/不可访问 | Codex 规则 |
| 语义解析通过 | 完整 ASR、角色证据、主题和边界 | 关键词单点、缺上下文、错配 | 规则＋人工抽检 |
| 动作配对通过 | 同动作、部位/动作名一致、循环完整、无跳题 | 普通手势、错部位、错动作、断裂 | 规则＋人工 |
| 候选切片通过 | 完整结构、自然切点、来源可追溯 | 半句话、半动作、主题跳转 | 人工审核 |
| 剪辑师直接可用 | Route A 硬闸门＋经验证阈值 | 仍需大量内容修正 | 用户确认阈值 |
| 人工审核通过 | 当前 candidate_version 明确 approved | 旧版本、无审核人、无意见 | 指定审核人 |
| 审美/人感通过 | 实际观看确认 | 生硬、节奏差、表达不自然 | 业务指定人员 |
| 业务通过 | 合规、品牌、健康、转化目标确认 | 夸大、错误优惠、隐私、无价值 | 业务负责人 |
| 正式交付通过 | 访问、来源、审核、接收确认 | 仅上传成功、链接过期 | 交付负责人 |
| 批量稳定运行 | 多批次、失败率、人审负担和恢复证据 | 单样本/遗漏失败 | 用户/产品负责人 |

## 9. required_inputs 总表

### 9.1 运动型

- 非专用代表素材、完整 ASR、视觉证据、动作术语、人工抽检和健康/业务规则。

### 9.2 口播型

- 纯口播金标准、完整 ASR、语义分类/切点样本、审核人和业务风险规则。

### 9.3 mixed

- 人工时间段路由标准、允许/禁止跨段规则和分段审核人。

### 9.4 飞书试点

- 管理员、测试应用、最小权限、试点用户、四类测试文件、批准存储与回调接收方式。

| 输出类 | 必需内容 |
|---|---|
| 视频 | review_proxy、final_clip、技术探针和访问策略 |
| 索引 | source、route、analysis、candidate、final 的主键与版本 |

## 11. 后续 Codex 分阶段工作包

每个阶段都应单独下发，禁止一次性建设整个产品：

1. schema 和状态机工作包。
2. 路由金标准与最小实现工作包。
3. Route A 通用化与回归工作包。
4. Route B 语义金标准与最小实现工作包。
5. 人工审核版本/反馈工作包。
6. 飞书账号 probe 工作包。
7. 飞书试点实现工作包。
8. 统一整合工作包。
9. 批量验证工作包。

每个工作包必须重新给出 P0、P1、allowed files、forbidden files、probe、失败标准和 Git 闭环。

## 12. validation_commands 规划

| 命令类型 | 中文用途 |
|---|---|
| schema lint | 验证实体、枚举和版本引用 |
| unit tests | 验证路由、语义、动作和状态硬规则 |
| golden set evaluation | 与人工金标准比较 |
| idempotency replay | 验证任务和回调重放 |
| integration dry run | 不调用外部 API 的步骤编排检查 |
| local callback mock | 验证 3 秒响应、重复事件和异步处理 |
| storage probe | 真实试点时验证上传、预览、权限、过期和撤权 |
| media probe | ffprobe 与完整解码 |
| safety scan | 密钥、媒体、缓存和临时 URL 排除 |
| git checks | diff check、限定路径暂存、push 和 remote readback |

## 13. blocked_if_missing

- Phase 1 路由合同未通过就开始双路线批量实现。
- Route A 缺非专用样本，Route B 缺人工语义金标准。
- 未统一 candidate_version 和 review_task 就接飞书。
- 未确认管理员、权限、存储和安全边界就创建正式应用。
- 未确认批量指标就自行上线。
- 任何阶段要求 LangGraph、LangChain、复杂 runtime 或未经批准数据库。
- 无法保证媒体、ASR 缓存和 secret 不进入 Git。
