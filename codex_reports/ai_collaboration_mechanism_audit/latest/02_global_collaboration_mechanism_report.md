# 02 全局协作机制报告

## 1. 项目协作总原则

这套机制库最核心的协作逻辑是：

> GPT 先判断，Codex 后执行，Perplexity 供资料，Obsidian 存机制，用户做最终方向 / 审美 / 业务裁决。

用户真正需要的不是“AI 帮忙多做一点”，而是一套可以长期复用的“项目配合机制操作系统”。这个系统要防止三类跑偏：

- 判断没做完就执行。
- 局部技术通过被写成整体完成。
- 旧项目经验、历史记忆、外部资料污染当前项目事实。

### ChatGPT 的角色

ChatGPT 是外层总控脑 / planning / critique / continuity 层，负责：

- 判断用户真实目标。
- 区分 P0 / P1 / P2。
- 补齐目标层、机制层、Implementation design、流程层、判断标准层、反馈层。
- 把方向型输入翻译成可执行合同。
- 生成 Codex 执行单。
- 复审 Codex 回报是否真的满足用户目标。

### Codex 的角色

Codex 是 execution / verification 层，负责：

- 在明确 workspace 和 allowed files 内读文件、改文件、跑命令、验证、落库。
- 做 impact check、dirty check、must read。
- 按已锁定 Implementation design 执行，不自行决定核心路线。
- 汇报证据，不夸大状态。
- 缺关键字段时 blocked。

### Perplexity 的角色

Perplexity / Web / 外部文档是研究层，负责：

- 查外部资料、官方文档、竞品信息、参考来源。
- 输出来源和可核查材料。
- 不直接把搜索摘要变成项目事实。

### Obsidian 的角色

Obsidian 是沉淀层，负责：

- 保存机制库、个人工作法、复盘、模板、长期知识。
- 不替代项目仓库、GitHub、当前文件、验证输出等 P1 事实源。

### 必须先判断的事情

- 本轮到底是闲聊、判断、机制设计、只读审计，还是执行任务。
- 用户表面问题和真实卡点是否一致。
- P0 / P1 / P2 是否冲突。
- 是否缺真实意图、Implementation design、验收标准、blocked 条件。
- 失败后应该回目标层、机制层、实现设计层、流程层、执行层、权限层还是验收层。

### 可以直接执行的事情

- 目标、边界、路径、允许文件、禁止事项、验证命令都清楚的局部任务。
- 只读读取和盘点。
- 已有执行单且 Implementation design 完整的文件修改 / 文档导出 / 工具链检查。
- 低风险、可回滚、范围明确的补全动作。

## 2. 用户输入处理机制

### 闲聊类

处理方式：直接回应，但不自动升级成项目事实或执行任务。

判断点：

- 是否只是情绪表达、想法整理、轻量讨论。
- 是否不需要落库、不需要 Codex、不需要外部检索。

输出方式：先结论，少术语，可以给一个下一步，但不强行推进。

### 外层总控判断类

典型输入：“方向可能有问题”“值不值得继续”“到底是 Codex 执行错了还是 prompt/spec 给错了”。

处理方式：

- 先进入真实意图澄清闸门。
- 判断本轮真正要证明什么。
- 不直接让 Codex 做 demo 或修代码。

输出方式：

- 本轮真正要判断。
- 本轮不判断。
- 成功 / 失败 / 停止条件。
- 需要的证据。
- 是否进入六层需求确认。

### 项目推进类

典型输入：“让 Codex 做一下”“把这个落到仓库”“生成执行单”。

处理方式：

- 必须先确认 P0/P1/P2。
- 必须补齐 Implementation design。
- 必须写清 Goal、Context、Constraints、Impact check、Execution steps、Done when、Blocked if、Output。

输出方式：Codex 执行单，而不是泛泛建议。

### 账号 / 工具 / 安全修正类

处理方式：

- 先判断是否涉及权限、API、余额、账号、secret。
- 不猜测、不暴露、不硬做。
- 如果缺权限或凭证，进入 blocked。

注意：机制库中只出现“旧账号/API key/余额”等规则描述，未发现实际 secret。

### 资料带回类

典型输入：用户从 Perplexity、网页、视频、截图、竞品资料带回内容。

处理方式：

- 先做外部资料保真提取。
- 拆成原始材料、保真事实、执行桥接、推测判断。
- 推测必须标 `推测`，真实性不足标 `待验证`。

输出方式：

- 可观察事实。
- 不可迁移内容。
- 可执行字段。
- 验收标准。
- 禁止项。

### “不对 / 空 / 差点意思 / 不顺 / 怪怪的”反馈类

处理方式：

- 不直接调参数。
- 先判断失败属于目标层、机制层、实现设计层、流程层、执行层、权限层还是验收层。
- 如果是审美/原感问题，回到原感参考层和执行桥接层。

输出方式：

- 失败现象。
- 失败证据。
- 失败层级。
- 保留 / 放弃 / 降级。
- 新 route / fallback / blocked 条件。

## 3. ChatGPT 应该怎么配合

### 回答前先判断表面问题和真实卡点是否一致

ChatGPT 不能只按用户话面执行。很多输入表面是“做一个东西”，真实卡点可能是：

- 项目是否成立。
- 机制是否缺口。
- 工具链是否够。
- Codex 是否缺上下文。
- 审美标准是否未锁定。
- 外部资料是否还没桥接。

### 先结论，少术语，能推进

用户偏好不是泛泛解释，而是：

- 先主结论。
- 再说明最小必要依据。
- 最后给能推进的下一步。

术语可以保留，但必须贴着项目使用，例如：目标层、机制层、流程层、判断标准层、反馈层、外层总控脑、Codex 默认不知道上下文。

### 什么时候必须拦截

- 真实目标不清。
- 用户输入同时包含方向判断和执行请求。
- 缺 Implementation design。
- 缺 primary_route / fallback_route / capability boundary / probe decision。
- 权限、API、余额、账号、安全信息不清。
- 外部资料还没有保真提取。
- 用户反馈“不对 / 怪怪的”，但失败层级没判断。

### 什么时候直接给下一步

- 用户只是要局部建议或轻量判断。
- 路径、目标、边界、验证方式都清楚。
- 本轮是只读审计或文档整理，不涉及不可逆动作。
- 已有完整执行单，只需交给 Codex。

### 什么时候给 Codex prompt

只有当以下内容齐备时：

- Goal 目标。
- Context 上下文。
- Constraints 边界。
- 真实意图澄清。
- Implementation design。
- Impact check。
- Must read。
- Execution steps。
- Done when。
- Blocked if。
- Output。

### 什么时候把 Perplexity 资料收束成执行中间层

当资料要进入项目推进时，ChatGPT 不能只摘要，而要转成：

- 保真事实。
- 可执行字段。
- 禁止复制项。
- 验收标准。
- `推测` / `待验证` 标记。

## 4. Codex 应该怎么配合

### Codex 默认不知道上下文

这是机制库里最重要的执行假设之一。GPT 聊天、Perplexity 搜索、用户口述、Obsidian 机制，都不会自动变成 Codex 当前上下文。每次下发必须显式写清。

### 每次下发必须包含

- 目标。
- 上下文。
- 边界。
- 影响面。
- 必须读取的文件。
- 执行步骤。
- 完成标准。
- 阻断条件。
- 输出格式。

如果任务涉及方向、能力、审美、外部资料，还必须包含 Implementation design。

### 执行前必须做

- workspace check。
- dirty check。
- allowed files / forbidden files check。
- must read。
- route_decision。
- implementation_design 完整性检查。
- 必要时先 probe。

### Codex 禁止行为

- 不允许脑补用户项目意图。
- 不允许随意重构或扩大影响面。
- 不允许把局部修复改成全局重写。
- 不允许把技术 demo 写成正式交付。
- 不允许把 local-only 写成 completed。
- 不允许把远端未验证写成 remote completed。
- 不允许把用户未人审写成审美/内容通过。

### Codex 应该 blocked 的情况

- 缺真实目标：`blocked_missing_true_intent_gate`。
- 缺 Implementation design：`blocked_need_implementation_design_layer`。
- 缺源文件、权限、路径、验证证据。
- P0/P1/P2 无法区分。
- 外部资料未提供到本轮输入或项目事实源。
- 继续执行会扩大边界。

## 5. Perplexity 应该怎么配合

### 负责的检索类型

- 官方文档、SDK、API、平台规则。
- 行业资料、竞品、案例。
- 外部参考视频、网页、素材来源说明。
- 用户需要快速判断方向时的资料收集。

### 回来后 ChatGPT 如何保真提炼

ChatGPT 要把 Perplexity 输出拆成：

- 原始来源。
- 可观察事实。
- 来源可靠性。
- 可执行字段。
- 不能迁移 / 不能复制的内容。
- 推测和待验证项。

### 可以进入项目知识的内容

- 稳定规则。
- 来源清楚的事实。
- 被验证过的工具能力。
- 抽象后的方法论。
- 与项目当前事实源一致的资料。

### 只能作为临时参考的内容

- 没有来源的摘要。
- 未核实的经验判断。
- 竞品 UI、品牌资产、可识别素材。
- 一次性搜索结论。
- 与当前项目文件冲突的外部资料。

## 6. Obsidian 应该怎么配合

### 机制库如何记录

Obsidian 适合保存：

- AI 分工模型。
- P0/P1/P2。
- 真实意图闸门。
- 六层需求确认。
- Codex 执行单模板。
- 状态词。
- blocked 规则。
- 外部资料保真方法。
- 失败后路线重判。

### 项目复盘如何沉淀

项目复盘应该拆成：

- 发生了什么。
- 证据是什么。
- 属于哪一层失败或成功。
- 可复用机制是什么。
- 不可迁移项目事实是什么。

### 适合进长期机制的内容

- 跨项目都成立的方法。
- 防跑偏规则。
- 执行单模板。
- 状态词和验收边界。
- 反馈闭环。

### 只适合进项目日志的内容

- 当前路径。
- 当前素材。
- 当前模型选择。
- 当前任务状态。
- 当前候选结果。
- 当前账号/API key/权限情况。
- 某次视频或业务验收结论。

## 7. 下发前确认机制

### 必须先确认需求的情况

- 用户目标、成功标准、失败标准、停止条件不清。
- 输入里同时包含方向判断和执行请求。
- 涉及项目主线、工具选型、继续投入。
- 涉及视觉、声音、剪辑、BGM、调色、自动化、API、候选产物。
- 用户反馈“不对 / 怪怪的 / 差点意思”。

### 可以自动补全的情况

- 缺少的是非关键格式字段。
- 任务是只读审计、盘点、整理、报告。
- 路线、边界、验证方式已经明确。
- 自动补全不会改变目标或扩大影响面。

### 必须阻断的情况

- 缺真实意图。
- 缺 Implementation design。
- 缺 workspace 或 allowed files。
- 缺关键来源文件。
- 权限/API/余额/账号条件不足。
- 无法区分机制和事实。
- 继续执行会把旧项目事实迁入新项目。

### 只做建议不执行的情况

- 用户仍在判断方向。
- 任务属于外层总控判断。
- 缺人审或业务判断。
- 外部资料未保真。
- 用户明确要求 read-only / analysis-only / mechanism-only。

## 8. 目标 / 机制 / 流程 / 判断标准 / 反馈闭环

### 目标层

解释：本轮真正要达成什么，是总目标、阶段目标，还是最小目标；本轮不解决什么。

视频项目用法：先判断是要做正片、样片、probe、风格拆解，还是只读参考分析。

直播项目用法：先判断是验证销售张力、数字人表现、TTS 文案、前台 MVP，还是机制复盘。

Codex 任务用法：Goal 必须写清任务类型和本轮不是。

### 机制层

解释：什么时候触发、什么时候禁止、什么时候降级、哪些能力仍是 `待验证`。

视频项目用法：比如参考视频只能抽象风格规则，不能复制素材。

直播项目用法：比如工具权限、API、余额失败时必须 blocked。

Codex 任务用法：明确 allowed_codex_autonomy 和 forbidden_codex_guessing。

### Implementation design（实现设计层）

解释：锁 primary_route、fallback_route、capability_status、probe_required、blocked_if_missing。

视频项目用法：先决定是 reference analysis、component probe、technical sample，还是 full render。

直播项目用法：先决定是文案机制、TTS 生成、前台验证，还是数据回填。

Codex 任务用法：缺这一层不执行，返回 `blocked_need_implementation_design_layer`。

### 流程层

解释：执行顺序、多候选、最小片段测试、用户回审。

视频项目用法：先小片段 / 接触表 / probe，再决定是否正片。

直播项目用法：先验证文案和音频，再做前台联动或发布动作。

Codex 任务用法：Execution steps 只能执行已锁路线，不能临时决定路线。

### 判断标准层

解释：技术通过、内容通过、审美通过、人感通过、失败标准分开。

视频项目用法：render 成功不等于高级感通过；技术样片不等于可发布。

直播项目用法：TTS 生成不等于销售张力成立；前台跑通不等于业务转化成立。

Codex 任务用法：validation、commit、push、remote readback、人审分别汇报。

### 反馈层

解释：方向错回目标层 / 机制层；路线错回实现设计层；执行粗糙回流程层；审美不清补 reference；权限失败 blocked。

视频项目用法：“怪怪的”先分层，不直接重 render。

直播项目用法：“不带货 / 不抓人”先判断是目标、文案、TTS、前台还是数据问题。

Codex 任务用法：失败后输出失败层级、证据、新 route 或 blocked。

## 9. 质量验收机制

### 可用

有明确入口、字段、产物路径、执行命令和输出格式。

### 好用

用户能看懂，ChatGPT 能下发，Codex 能执行，失败能回到正确层。

### 稳定

不是单次 probe；有多轮或当前证据支持；能力状态不夸大。

### 可复用

只保留机制，不夹带旧项目路径、素材、模型、状态、业务结论。

### 可解释

每条判断能说明来源：P0 用户本轮输入、P1 当前项目事实、P2 参考层。

### 可回滚

Codex 改动范围小、reviewable、可验证；Git 状态和远端 readback 分开。

### 能人工兜底

审美、人感、业务目标必须保留用户裁决，不让 AI 技术结果替代。

### AI 和人工分工清楚

GPT 判断和复审，Codex 执行和验证，Perplexity 查资料，Obsidian 沉淀机制，用户裁决方向和质量。

## 10. 冲突、重复、过期机制清单

### 互相冲突

- 已确认：没有发现明显互相冲突的机制。
- 待验证：未来如果把这些规则写入不同 GPT Project，可能出现“项目本地 AGENTS.md”和“通用机制库”冲突；按 P0 > P1 > P2 处理。

### 重复规则

- “GPT 先判断，Codex 后执行”在 `00`、`01`、`04`、`05`、`99` 中重复出现，是核心重复。
- “缺 Implementation design 必须 blocked”在 `00`、`03`、`04`、`05`、`06`、`12` 中重复出现，是关键防跑偏规则。
- “未验证写待验证，不夸大 completed”在 `02`、`10`、`99` 中重复出现，是完成度硬规则。
- “只迁移机制，不迁移项目事实”在 `00`、`01`、`07`、`08`、`11`、`12`、`99` 中重复出现，是跨项目迁移规则。

### 可能过期

- 已确认：未发现文件标记为 deprecated。
- 推测：如果后续已经有新的 GPT Project Instructions 或 AGENTS.md，这 14 篇仍需要与最新项目规则对齐。

### 未完成 / 草稿感

- `12_常用Codex执行单模板库` 中模板的 `Impact check`、`Must read`、`Execution steps`、`Done when`、`Blocked if` 多处是留白模板，不能当成完整任务单直接给 Codex。
- `05_Codex执行单标准模板` 也是标准框架，不是具体任务执行单。

### 需要用户确认后再进入 GPT Project 的规则

- Codex 是否默认必须 commit + push + remote HEAD verification 才算完成：机制库强调分层汇报，但具体项目是否每次都要 push 需要按项目确认。
- 哪些任务允许 Codex 自动补全，哪些必须先问用户：机制库给了原则，但不同项目的风险阈值要确认。
- Perplexity 资料是否进入 Project Knowledge：需要用户确认来源可靠性和版权/品牌边界。
- 是否把 `12` 的全部模板放入 Project Knowledge：建议放，但不建议作为 Instructions。
